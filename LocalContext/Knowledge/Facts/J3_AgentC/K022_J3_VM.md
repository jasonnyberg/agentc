# Knowledge: J3 AgentC VM

**ID**: LOCAL:K022
**Category**: [J3 AgentC](index.md)
**Tags**: #j3, #vm, #bytecode, #stack-based

## Overview
The AgentC VM (`EdictVM`) is a C++ implementation of the Edict virtual machine, designed to operate on the unified `Listree` data structure. It executes bytecode that manipulates five primary resource stacks, facilitating a "Stack of Lists" execution model.

## Resource Stacks
The VM maintains state in `resources[VMRES_COUNT]`, where each resource is a `ListreeValue` (List):
1.  `VMRES_DICT`: Dictionary/Scope stack.
2.  `VMRES_STACK`: Data stack.
3.  `VMRES_FUNC`: Function/Call stack.
4.  `VMRES_EXCP`: Exception stack.
5.  `VMRES_CODE`: Code stack (Instruction Pointers).

## Instruction Set
- **Stack Ops**: `DUP`, `SWAP`, `POP`, `CONCAT`, `LIST_ADD`.
- **Navigation/Var**: `REF` (Resolve), `ASSIGN` (Store), `REMOVE` (Delete), `EXT` (Extract Literal).
- **Execution**: `EVAL` (Execute TOS), `YIELD`.
- **Frames**:
  - `FRAME_PUSH`: Pushes a new isolated stack frame.
  - `FRAME_MERGE`: Merges the top stack frame into the previous one (concatenation).
- **Context/Scope**: `CTX_PUSH`/`CTX_POP` (Manage Dictionary frames).
- **Functions**: `FUN_PUSH`/`FUN_EVAL`/`FUN_POP` (Manage Function frames).
- **Shuffles**: `S2S`, `S2D`, `S2E`, etc. (Move items between resources).

## Execution Model
- **Bytecode**: Executed sequentially via `execute()`.
- **Term Rewriting**: Experimental support for rule-based rewriting of the stack before/during execution.
- **Cursor**: The VM owns a `j3::Cursor` to track its position in the Concept Space (though mostly used for debugging/visualization in this context).

## Comparison to J2
- **Frames**: J3 introduces `FRAME_PUSH`/`FRAME_MERGE` explicitly to support the `f(x)` isolation semantics (where arguments are evaluated in a temp frame).
- **Rewriting**: J3 is designed with Term Rewriting as a first-class citizen (System 2 thinking).
- **Implementation**: C++ class based, using `CPtr` for safe resource management.
