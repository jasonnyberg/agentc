# J2 Project Analysis

## Overview

J2 is a concatenative programming language designed to understand the structure of and dynamically link with compatible C binaries. The project implements a unique data structure called "LisTree" (a combination of a list and tree) and a cursor-based navigation system called "REF" that allows for symbolic access to objects maintained in the tree structure.

## Key Components

### LisTree Structure

The LisTree is a hybrid data structure that combines:

1. **AA Tree**: An implementation of Arne Andersson's balanced binary search tree for efficient lookups
2. **Circular Linked Lists (CLL)**: For maintaining collections of values

The core components of the LisTree structure are:

#### LTV (LisTree Value)
```c
typedef struct {
    union {
        CLL ltvs;
        LTI *ltis;
    } sub;
    LTV_FLAGS flags;
    void *data;
    int len;
    int refs;
} LTV;
```

#### LTI (LisTree Item)
```c
struct LTI {
    LTI *lnk[2]; // AA TREE LEFT/RIGHT
    char *name;
    CLL ltvs;
    unsigned short len;
    unsigned char level, preview[PREVIEWLEN];
};
```

#### LTVR (LisTree Value Reference)
```c
typedef struct {
    CLL lnk;
    LTV *ltv;
} LTVR;
```

### REF Structure (Cursor Implementation)

The REF structure is the "cursor" class that implements state management for traversing the LisTree structure. It acts as a micro-interpreter that allows symbolic access to objects maintained in the tree.

```c
typedef struct REF {
    CLL lnk;
    CLL keys;   // name(/value) lookup key(s)
    CLL root;   // LTV being queried
    LTI *lti;   // name lookup result
    LTVR *ltvr; // value lookup result
    LTV *cvar;  // cvar deref result
    int reverse;
} REF;
```

## REF Operations

The REF structure provides a powerful cursor-based navigation system with the following key operations:

### Creation and Initialization
- `REF_init`: Initialize a REF structure
- `REF_create`: Create a new REF from a string representation
- `REF_HEAD/REF_TAIL`: Get the head or tail REF from a collection

### Navigation and Resolution
- `REF_reset`: Reset a REF to a new root
- `REF_resolve`: Resolve a REF path against a root LTV
- `REF_iterate`: Iterate to the next matching item

### Manipulation
- `REF_assign`: Assign a value to the location referenced by a REF
- `REF_replace`: Replace a value at the location referenced by a REF
- `REF_remove`: Remove the item referenced by a REF

### Access
- `REF_lti`: Get the LTI referenced by a REF
- `REF_ltvr`: Get the LTVR referenced by a REF
- `REF_ltv`: Get the LTV referenced by a REF
- `REF_key`: Get the key used by a REF

## Micro-Interpreter Functionality

The REF structure acts as a micro-interpreter by:

1. **Parsing Path Expressions**: Breaking down path expressions into components
2. **Resolving Symbols**: Looking up symbols in the tree structure
3. **Navigating the Tree**: Moving through the tree based on the path
4. **Manipulating Data**: Allowing modification of the tree structure

## Implementation for J3

To implement a similar cursor class in the J3 project:

1. **Define a Cursor Class**: Create a class that maintains state for traversing the AATree structure
2. **Implement Path Resolution**: Add functionality to resolve symbolic paths in the tree
3. **Support Navigation Operations**: Implement methods for moving through the tree
4. **Add Manipulation Methods**: Create methods for modifying the tree structure
5. **Ensure Memory Safety**: Leverage the stack-based allocation model for memory safety

## Proposed Cursor Class for J3

```cpp
template<typename T>
class Cursor {
private:
    CPtr<AATree<T>> root;      // Root of the tree being navigated
    CPtr<AATree<T>> current;   // Current position in the tree
    std::string path;          // Current path expression
    bool reverse;              // Direction flag

public:
    // Prevent heap allocation
    void* operator new(size_t) = delete;
    void* operator new[](size_t) = delete;
    void operator delete(void*) = delete;
    void operator delete[](void*) = delete;

    // Constructor
    Cursor(CPtr<AATree<T>> root);
    
    // Navigation
    bool resolve(const std::string& path);
    bool next();
    bool prev();
    bool up();
    bool down();
    
    // Access
    CPtr<AATree<T>> getNode() const;
    CPtr<T> getValue() const;
    std::string getName() const;
    
    // Manipulation
    bool assign(CPtr<T> value);
    bool remove();
};
```

The diagram above illustrates the hierarchical organization of the J2 system. At the core are the fundamental data structures, primarily the Listree, which serves as the foundation for the dictionary, data stack, and code stack. The Edict language parser/compiler translates source code into bytecode, which is then executed by the virtual machine.

The reflection system processes DWARF debugging information from shared libraries to build a type database, which enables the FFI system to make calls to C functions. The bootstrap engine initializes the system and provides the basic Edict functions needed for operation.

All components interact through well-defined interfaces, with the Listree serving as the common data representation format throughout the system. This architecture allows J2 to seamlessly bridge the gap between the static world of compiled C code and the dynamic world of interpreted languages.
